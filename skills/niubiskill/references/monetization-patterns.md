# 赚钱模式候选库

这不是答案库。先看项目底牌，再用这些模式生成最多三个候选；没有匹配条件就不要推荐。

| 模式 | 谁付钱 | 对方买什么 | 收钱事件 | 常见起步路线 | 成立前提 |
|---|---|---|---|---|---|
| 一次性产品 | 最终使用者或采购方 | 一个标准化结果 | 下单 / 付款 | 引流或成交 | 产品边界、价格和交付清楚 |
| 产品化服务 | 有明确问题的个人或组织 | 固定范围、周期和结果 | 订金 / 项目款 | 成交 | 能控制范围、修改和工时 |
| 付费诊断 | 问题昂贵但方案尚不清楚的一方 | 判断、路线和风险边界 | 预约付款 / 诊断费 | 成交 | 诊断本身有独立价值，不是伪装售前 |
| 订阅 / 会员 | 持续获得价值的使用者或组织 | 连续访问、更新、服务或身份 | 周期扣款 / 续费 | 引流后成交 | 价值持续发生，流失原因可处理 |
| 按量计费 | 使用频率可记录的一方 | 实际使用量 | 调用量 / 席位 / 交易量 | 成交 | 计量可信、边际成本可控 |
| 许可 / 授权 | 需要合法使用资产的组织 | IP、品牌、内容、技术或区域权利 | 许可合同 / 周期费用 | 成交 | 权属清楚、使用范围可执行 |
| 免费核心 + 企业服务 | 企业预算方 | 私有部署、治理、集成、培训或责任 | 项目费 / 支持费 | 成交 | 免费核心完整，企业需求真实不同 |
| 佣金 / 推荐 | 获得合格交易或客户的一方 | 筛选、匹配、介绍或成交结果 | 合格线索 / 成交 / 结算 | 引流或成交 | 双方知情、归因清楚、持续价值和权限成立 |
| 平台抽成 | 供需一侧或双方 | 交易、信任、结算和治理 | 每笔交易 / 服务费 | 引流 | 两侧价值成立，平台不只靠阻断联系 |
| 广告 / 赞助 | 想触达特定人群的品牌 | 合格注意力、内容或活动权益 | 发布 / 曝光 / 合格线索 | 引流 | 受众明确、披露充分、注意力质量可证明 |
| 培训 / 认证 | 个人、雇主或生态伙伴 | 能力提升、实施方法或可信证明 | 报名 / 认证费 | 引流后成交 | 有可检验课程结果和合格交付 |
| 数据 / 报告 | 需要持续判断的组织 | 合法、独特、及时的洞察 | 订阅 / 报告 / API费用 | 成交 | 数据权利、质量、隐私和更新成本清楚 |

## 匹配规则

1. 先找已经存在的底牌，不从零编造产品、受众和渠道。
2. 优先匹配已经出现询问、使用、付款或重复需求的模式。
3. 一个候选只保留一种主要收钱事件。
4. 广告、佣金和数据模式必须先核受众质量、归因、披露、权利和监管边界。
5. 订阅不是“一次性产品按月收钱”；必须有持续价值。
6. 平台不是“建一个网站让双方自己来”；必须解决交易、信任、结算或治理问题。
7. 免费核心商业化不靠故意阉割免费版本，而靠企业特有责任和服务。

没有当前来源和真实证据时，标记为`假设`或`待核`，不得写成“已验证赚钱模式”。
