# NIUBI Skill

把项目丢进来，找出离钱最近的那个点。

> **获客 + 成交 = 赚钱。**

## 结果

每次只交付：`一个赚钱点 + 一条路线 + 一个本周动作 + 一份可用材料 + 一个七天证据`。

后台最多比较三个候选，前台只展开赢家。不要输出商业计划、底牌表、候选表、评分表、完整思维过程或两套方案；能一句说清，不写一段。

## 输入

网址、一句话、文件或现状都可以。先用已有输入和获准查看的材料；只有一个缺口会改变付款方、商品、路线或权限时，才问一个问题。

关键事实标记为`已核验 / 用户自述 / 待核 / 假设`。用户输入默认是`用户自述`；只有本轮看过的原件、系统记录或当前官方来源才是`已核验`。状态贯穿全文，推理和改写不能升级证据。用户未提供或本轮未核验的付费理由、需求与交付能力必须保留为假设；进入对外材料时删除或写成条件句。

## 后台三步

### 1. 锁定收钱闭环

赚钱点必须说清：`谁付钱 + 为什么付 + 卖什么 + 什么事件触发收入`。缺任何一项，只是想法。

读取 [monetization-patterns.md](references/monetization-patterns.md)，从不同收钱逻辑中生成最多三个候选；只有一个清楚时不要硬凑。排序固定为：

`权限可做 → 已有需求证据 → 离收钱近 → 七天可验证 → 能交付且经济性可能成立`

ABCD只定位瓶颈：缺合格线索看`A / 引流`，有线索无承诺看`B / 成交`，交付、复购或产品基础是瓶颈时直接修瓶颈。多方角色会改变结论时，再读取 [commercial-loop.md](references/commercial-loop.md)。

### 2. 过闸门并选路线

涉及雇佣冲突、第三方资料、数据、IP、平台、跨境、受监管活动、未成年人或收益宣传时，读取 [evidence-and-boundaries.md](references/evidence-and-boundaries.md)。闸门判断的是`下一步具体外部动作`：

- 缺线索：`引流`，材料带一个明确 CTA；
- 有目标对象或询问但无承诺：`成交`，材料请求一个有成本承诺；
- 当前动作的必要前提若为否会导致违法、未授权、误导、无法履约、泄露或不可逆风险：`暂停`，材料只用于取得权限或证据；
- 已知违规、欺骗、侵权或规避：拒绝原动作，只给合法替代。

只影响以后收款或交付的事项不阻断当前安全动作，但写明最晚核清节点。改名或换包装不能消除原依赖。

### 3. 交付材料与七天证据

材料必须是完整成品，只服务本周动作。播放、点赞、泛关注和客套回复不算钱；提交真实资料、接受价格、投入决策时间、发起采购、订金、订单、复购等才是前进证据。

测试固定一个目标角色、一个入口、一个动作和一个合格事件，并写清有效触达与保存证据：

- 事件出现：只说明这个条件下出现过证据，进入下一轮；
- 事件没出现：有效触达后只改一个变量；未有效触达则写`证据不足`；
- 权限失败、无法履约或真实单位经济不成立：停当前动作或方案。

一次`0`或`≥1`都不能证明整个市场、商业模式或项目成立或死亡。再次运行时优先使用上轮真实结果，但只把它当作带日期与场景的观测。

## 数字与断言

影响报价、对外承诺、工作量/成本或第七天决定的关键商业数字，首次出现时必须四选一：

- `证据值`：来源 + 日期（若会过时）+ 证据状态；
- `推导值`：最短算式，继承最弱输入状态；
- `实验参数`：说明本轮约束或理由，不冒充事实、预测或行业基准；
- `未知`：删除，或仅在材料必需时用`【占位符】`。

无依据区间仍无依据。没有真实基线、硬约束或单位经济时，实验参数未命中不能判死项目。对外材料不得编造原价、库存、稀缺、成品、能力、案例、客户反馈或效果承诺；只有出现占位符时才显示发出前自查。提交前重算公式与时间线，同一数字不得变值。
