# NIUBI Skill

**当使用者、付款者、决策者不是同一个人，先把商业系统画清楚。**

学校教知识；NIUBI Skill 训练你把已有的能力、关系、渠道和信息组织成可验证的商业闭环。

它会映射最多 5 个关键角色，以及价值、钱、权力、信息四条流；找出你当前能够正当、经授权地捕获价值的位置；最后只给一个 ABCD 七天承诺实验和一个可直接使用的行动资产。

为全球华语创业者、客户经理、渠道 / BD、创作者和开源作者设计。

> 不承诺收入，不把“认识人”当作价值，也不把口头兴趣写成成交。

## 30 秒开始

Codex 一行安装（需要 Node.js 18+）：

```bash
npx skills add nathanskill/niubiskill --skill niubiskill -g -a codex -y
```

或者手动克隆：

```bash
git clone https://github.com/nathanskill/niubiskill.git
cd niubiskill
```

将 `skills/niubiskill` 复制到 Agent 的 Skill 目录。Codex 默认目录示例：

```bash
mkdir -p ~/.codex/skills
cp -R skills/niubiskill ~/.codex/skills/niubiskill
```

重新打开 Agent 后直接调用：

```text
$niubiskill

我是[角色]，已经拥有[产品/技能/关系/渠道]，想推动[具体结果]。
已知事实是[事实]，限制是[权限/时间/成本/地区]。
请只给我一张最小商业闭环卡、一个七天承诺实验和一个行动资产。
```

## 一个合成案例：客户经理不需要向客户本人收费

> 以下为跨行业合成案例，只用于展示方法，不代表任何真实客户、项目或结果。

林是一家库存管理 SaaS 的渠道客户经理。本地实施商有客户关系，却缺少产品技术；SaaS 公司有产品和工程师，却不了解当地交付环境。终端客户向公司和实施商付款，林通过雇佣合同获得工资与季度奖金，而不是向客户私下收费。

NIUBI Skill 会把这个局面压缩为：

- **一链：** 终端客户 ↔ 本地实施商 ↔ 林（渠道客户经理）↔ SaaS 公司。
- **四流：** 客户得到软件与实施；钱按合同流向公司和实施商；公司控制价格、合同和奖金归属；实施商、林和工程团队交换需求、可行性与交付反馈。
- **价值捕获依据：** 林筛选真实机会、形成可追踪归因、协调内部资源并降低多方交接成本；回报依据是雇佣合同和正式激励规则。
- **持续价值测试：** 即使公司和实施商可以直接沟通，林仍通过机会筛选、内部编排、归因和异常处理持续创造价值。
- **当前 ABCD：** `B — Bargaining`。节点已经出现，缺少公司对试点规则和资源的有成本承诺。
- **一诺：** 公司在七天内书面批准“1 家实施商 × 1 个合格机会”的 30 天试点，完成 CRM 归属并排期一名技术人员。
- **一个行动资产：** 一页内部试点批准请求。
- **证据：** 书面批准、CRM 记录和技术排期；没有这三项，就不把合作写成已推进。

## 你会得到什么

每次严格输出两个顶层块：

```text
A. 商业闭环卡
   一链：最多五个关键角色
   四流：价值 / 钱 / 权力与权限 / 信息
   一位：用户的主位置与价值捕获依据
   一环：当前唯一瓶颈关系

B. 七天承诺实验
   一诺：一个有成本承诺
   一入口：一个有权使用的接触点
   一资产：一份可以立即使用的成品
   一周：一次执行与结果回填
```

最终只做三种决定：继续、修改一个变量、停止。

## 什么时候使用

- 已经有技能、产品或服务，却不知道应该找谁、由谁付款；
- 是客户经理、渠道或经纪角色，不应该向终端客户本人收费；
- 免费用户很多，但开源项目、内容或社群没有清楚的企业付款方；
- 合作方都说“有兴趣”，却没有合同、资源配置、试点或其他真实承诺；
- 认识很多人、掌握行业信息，但没有稳定的价值贡献和分配机制；
- 获客、成交、交付、续用或持续产品弹药中只有一个环节真正阻塞。

## 它和普通销售提示词有什么不同

普通销售提示词通常默认“一个卖方 + 一个买方”。NIUBI Skill 先检查：

1. 受益者、使用者、决策者、付款者、分配者是否为同一主体；
2. 用户创造了什么价值，凭什么获得工资、奖金、服务费、分成、订阅、股权或其他回报；
3. 谁控制价格、客户、数据、结算、暂停和退出；
4. 即使参与者可以直接沟通，为什么仍愿意让用户参与；
5. 七天内哪一个承诺最能证明闭环真的前进。

## ABCD

- **A — Acquisition / 获客：** 获得当前闭环缺少的合格节点。
- **B — Bargaining / 成交：** 形成明确规则、责任、下一步和真实成本投入。
- **C — Customer Care / 维护：** 让交付、结算、信任、反馈和续用持续运转。
- **D — Development / 学习与资产沉淀：** 沉淀产品、工具、案例、数据、SOP、内容和伙伴能力。

它们不是一次性漏斗，而是循环：`D → A → B → C → D`。

## 边界

NIUBI Skill 不帮助：

- 保证赚钱、成交、收益或投资回报；
- 隐藏佣金、重大利益冲突或真实商业身份；
- 盗版、破解、刷量、假账号、骚扰或规避平台和监管；
- 未经同意分享联系人、客户材料、公司数据或第三方知识产权；
- 通过阻止合作方直接沟通来人为制造依赖。

遇到雇佣、跨境、数据、平台或受监管活动，Skill 会先做权限闸门。规则不清时只输出内部结构和待核清单，不生成外部招揽方案。
