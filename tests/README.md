# Tests

`scenarios.json` 是单轮行为验收规范，`conversations.json` 是连续对话验收规范；两者都不是模型跑分结果。每个场景写明用户输入、必须出现的行为和不得出现的行为。

这些场景只是少量回归探针，不是Skill允许处理的场景清单，也不规定唯一正确答案、固定标题、固定商业模式或固定动作。审计判断原则是否满足，不比较输出是否复刻示例。

`python3 scripts/validate_repo.py` 只检查仓库结构、Skill约束、示例长度和场景规范是否完整；它不会调用模型，也不代表这些行为已经通过。正式前向测试见`protocol.md`。

比较版本时，应让同一模型在同一输入、同一审计口径下分别生成结果，再独立检查：

- 第一段是否先给改变决策的暂定判断，而不是只返回问题；
- 用户未指定形式时是否突出主判断；用户要求比较、审计、访谈或发散时是否完整服从；
- 只有生产惯性确实挤占更强商业动作时，是否才明确叫停；
- 用户当前动作正确时，是否肯定并收窄，而不是为了格式强行反对；
- 推荐生产任务时，是否同时存在具体买方、明确成交阻塞、最小修复和修完后的承诺动作；
- 有旧客、询盘或可联系付款方时，是否优先成交而不是继续做内容或功能；
- 路径清楚时是否直接判断，路径不清时才比较不同价值捕获逻辑；
- 是否请求当下可行的最高级证据，并区分收钱证据与前进证据；
- 关键商业数字是证据值、推导值、实验参数还是未知；
- 假设是否被写成事实；
- 对外材料是否编造价格、能力、案例、稀缺或客户反馈；
- 闸门是否只约束当前动作；
- 输出长度与材料可用性；
- 用户补充信息或说“继续”时，是否只给新的变化，不重印整张卡或编造新颖感。

没有原始输出、统一口径和可复算记录时，不发布胜率或改进百分比。
