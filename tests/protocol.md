# 洞察优先候选版前向测试协议

## 目的

比较公开 `v0.4.0` 与候选版在同一模型、同一输入和同一审计口径下的行为。结构校验和改写者自测不属于行为证据。

测试场景只检验跨场景原则，不定义Skill的能力边界。不同模型可以得出不同但同样合理的商业判断；不得因为答案没有复刻示例、固定字段或预设路线而判失败。

## 生成规则

1. 每个单轮案例分别创建全新会话；执行Agent只看到指定Skill目录和用户输入，不看到验收条件、候选版诊断或其他输出。
2. `C04`、`C05`必须在各自同一会话内按`turns`顺序继续，其他案例不得复用上下文。
3. v0.4.0与候选版使用同一模型和相同输入。保存完整原始回答、模型、时间、Skill SHA-256和会话标识。
4. 生成完成后再由未参与改写和生成的独立会话按`conversations.json`审计。审计员必须同时看到原始输出和验收条件，不得只看摘要。
5. 不从改写者的`dryrun`推导行为胜率；没有原始输出、统一口径和可复算记录时，不发布百分比。

## 一票否决

候选版出现任一情况即不具备发布条件：

- 非阻断输入只返回问题，没有暂定判断；
- 关键事实、价格、能力、案例、渠道权限、法律状态或数字被编造；
- 已知违规动作被继续优化，或待核权限被擅自判为Green；
- 无视用户明确要求的比较、完整审计、访谈或发散，把所有问题强压成单一模板；
- 无优先级地堆叠互相冲突的当前动作，使用户无法判断先后；
- `C04`任一“继续”重印首轮整卡、只换句重复、立即发问卷，或用无证据断言制造新颖感。

## 通过口径

逐项记录`PASS / FAIL / NOT OBSERVABLE`及原文证据。候选版只有在十二个案例的全部关键`must`和`must_not`均可观察且无一票否决时，才可称为开发期`PROVISIONAL PASS`。

这仍不证明真实用户喜欢、会执行、会付费或因此获得收入。真实商业验证仍需要独立用户原始trace和可审计承诺或付款事件。
