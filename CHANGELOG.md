# Changelog

## Unreleased

## 0.4.0 — 2026-07-14

- 将产品升级为双核心“赚钱纠偏 + 获利路径”：先打断不产生商业证据的生产惯性，再从不同价值捕获逻辑中最多比较三个，只推荐一个；
- 扩展触发条件，在用户准备继续开发、发内容、改版、自动化、招人或追加投入但缺少付费证据时主动介入；
- 新增生产许可规则：只有具体买方提出明确成交阻塞、修复范围最小且修完立即推进承诺或付款时，才继续生产；已收款后的必要履约不被误停；
- 将商业链收敛为“可联系付款方、问题确认、明确报价、有成本承诺、到账/复购”，只推进第一个断点；
- 建立收钱证据与前进证据的强弱顺序，不再把播放、点赞、Star、会议或客套兴趣写成商业结果；
- 将获利模式候选库扩展为18类主要价值捕获逻辑，并为每类补充收钱事件、最小验证与成立前提；
- 重写README首屏、使用时机、证据分级与引流/成交说明，保持“一屏判断 + 一份直接用材料”；
- 将行为规范从17项扩展到25项，新增停止仪表盘忙碌、停止内容日历、停止第六个AI工具、允许真实买方阻塞下的最小开发、长采购周期、Amber去忙碌化、极简输入防编造和商品/付款方双未知硬闸门场景；
- 限制实验参数：不得凭空创造价格、人数、目标资格、渠道、联系人、产能或交付范围；阻塞动作时只问一个问题，否则保留占位符；
- 为极简输入增加硬闸门：当前可交付结果与合法触达对象都未知时，只问一个合并问题并停止，不再擅自把软件改造成咨询或人工服务；
- 加强仓库校验：验证v0.4输出合同、双核心规则、模式覆盖、稳定T01–T25 ID、场景字段类型，以及VERSION、CITATION日期和CHANGELOG标题一致性。

## 0.3.0 — 2026-07-13

- 将默认输出压成“结论、现在、本周、七天证据、直接用”，候选、底牌、ABCD、角色图和审计过程退到后台；
- 将核心 `SKILL.md` 从188行压到90行，三个示例从每个约100行压到约24–33行；
- 新增关键商业数字纪律，区分证据值、推导值、实验参数与未知，不再把诚实披露的任意参数伪装成证据；
- 禁止假设中途升级、无依据区间、虚构原价/稀缺/能力/案例/客户反馈，以及一次`0`或`≥1`判死整个项目；
- 将权限闸门收敛到下一步具体动作，不再因未来收款或交付事项待核而暂停安全研究；
- 同一项目再次运行时优先使用带日期与场景的真实结果，未出现证据只修改一个变量；
- 行为场景从11个增加到17个，覆盖数字对账、实验阈值、断言真实性、动作级闸门和跨轮校准，并明确仓库校验不等于模型跑分；
- 补充AI时代的变现判断、发起人经历、开源理由，以及对公开营销、销售、产品发现和Skill工程项目的设计取舍。

## 0.2.0 — 2026-07-12

- 将首次使用压缩为“一行安装—丢入项目—拿到赚钱点卡”三步，不再要求用户先填长表；
- 将产品收敛为单一功能“项目变现点挖掘器”；
- 用一张`NIUBI赚钱点卡`替换旧商业闭环卡与七天实验双块输出；
- 每次最多生成三个不同赚钱点，只推荐一个；
- 将当前动作明确分为`引流`与`成交`两条路线，权限问题保留`暂停`例外；
- 新增12类赚钱模式候选库和统一内容卡结构；
- 重写README、Agent入口、三个跨行业合成案例和十个行为场景；
- 保留ABCD、多角色四流和权限闸门作为按需后台能力。
